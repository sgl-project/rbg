## 按部署架构选择文档

部署架构（聚合 vs PD 分离）决定了运维复杂度，是选择文档的第二维度。

### 聚合部署（Aggregated）

推理引擎作为单一角色，部署和运维相对简单。

| 运维需求 | 推荐文档 | 说明 |
| --- | --- | --- |
| **首次部署** | 文档 1 | 选择单机或多机拓扑 |
| **滚动更新** | 文档 3 | 基础滚动更新配置 |
| **升级加速** | 文档 4 | 原地升级 + 原地调度 |
| **预热加速** | 文档 6 | 大模型建议预热 |
| **弹性伸缩** | 文档 8（HPA/KEDA） | 指标或事件驱动伸缩 |
| **KV Cache** | 文档 10（可选） | 如果需要跨 Pod KV Cache 复用 |

**不需要关注**：CoordinatedPolicy（单角色无需协调）。

---

### PD 分离部署（Disaggregated）

Prefill 和 Decode 作为独立角色，需要协调伸缩和升级。

| 运维需求 | 推荐文档 | 说明 |
| --- | --- | --- |
| **首次部署** | 文档 1（PD 分离拓扑） | 单机或多机 PD 分离 |
| **滚动更新** | 文档 3 | 每个角色的滚动更新策略 |
| **升级加速** | 文档 4 | 原地升级 + 原地调度 |
| **预热加速** | 文档 6 | 大模型建议预热 |
| **协作部署** | 文档 7（**必须**） | CoordinatedPolicy 控制首次部署的渐进式创建 |
| **协作升级** | 文档 7（**必须**） | CoordinatedPolicy 控制多角色同步升级 |
| **弹性伸缩** | 文档 8（RBG Planner，**强烈建议**） | PD 分离需要 SLA 驱动的预测式伸缩 |
| **KV Cache** | 文档 10（**强烈建议**） | Prefill 产生的 KV Cache 需要传递给 Decode |

## 典型场景推荐路径

### 场景 A：快速部署 Qwen-72B 推理服务

**目标**：在生产环境部署 Qwen-72B，提供稳定的推理服务。

**推荐路径**：

```plain
1. 阅读文档 1（聚合部署 + 多机多卡）
   ↓ 了解 LeaderWorkerPattern 拓扑

2. 阅读文档 6（Warmup 预热）
   ↓ 预热模型权重，缩短首次就绪时间

3. 阅读文档 3（滚动更新）
   ↓ 配置更新策略

4. 阅读文档 4（原地升级）
   ↓ 配置原地升级，避免重建时重新加载模型

5. 阅读文档 8（弹性伸缩）
   ↓ 配置 HPA/KEDA 进行弹性伸缩

6. 阅读文档 5（服务发现）
   ↓ 了解 DNS 和 ConfigMap 服务发现
```

---

### 场景 B：部署 DeepSeek-R1 PD 分离推理服务

**目标**：部署 DeepSeek-R1 671B 的 PD 分离架构，确保 Prefill 和 Decode 协调运行。

**推荐路径**：

```plain
1. 阅读文档 1（PD 分离 + 多机多卡）
   ↓ 了解 PD 分离拓扑和 LeaderWorkerPattern

2. 阅读文档 7（CoordinatedPolicy）
   ↓ 配置首次部署的渐进式创建和同步升级

3. 阅读文档 6（Warmup 预热）
   ↓ 预热 671B 模型权重和 DeepGEMM 预编译

4. 阅读文档 10（Mooncake Store）
   ↓ 部署分布式 KV Cache，Prefill → Decode 传递

5. 阅读文档 4（原地升级）
   ↓ 配置原地升级，保留 KV Cache

6. 阅读文档 8（RBG Planner）
   ↓ 配置 SLA 驱动的预测式伸缩

7. 阅读文档 5（服务发现）
   ↓ 了解 Prefill 和 Decode 之间的服务发现
```

---

### 场景 C：大规模集群 Controller 性能调优

**目标**：在管理 500+ RBG 实例的大规模集群中，确保 Controller 性能达标。

**推荐路径**：

```plain
1. 阅读文档 9（资源评估与压测）
   ↓ 了解 Controller 资源需求和配置指南

2. 运行 /stress-test Skill
   ↓ 执行压测，收集性能数据

3. 分析报告，调整参数
   ↓ 根据文档 9 的调优决策树调整

4. 重新压测验证
   ↓ 确认配置满足目标规模
```

---

### 场景 D：hostNetwork + RDMA 推理服务

**目标**：部署使用 hostNetwork 的 RDMA 推理服务，避免端口冲突。

**推荐路径**：

```plain
1. 阅读文档 5（端口分配与服务发现）
   ↓ 重点阅读第 3 层：端口分配与组件发现

2. 启用 Controller 端口分配器
   ↓ --enable-port-allocator=true

3. 配置端口分配注解
   ↓ PodScoped / RoleScoped 端口分配

4. 配置组件发现注解
   ↓ addressRefs + portRefs

5. 阅读文档 1（CustomComponentsPattern）
   ↓ 了解多组件部署拓扑
```
