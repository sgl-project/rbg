# Developer Guide

## How to release

### Helm Chart Version Updater  

**Script Path:** `rbg/tools/release/version-update.sh`  
**Purpose:** Automatically updates Helm chart versions and image tags based on Git branch names.  

---

### üöÄ Quick Start  

1. **Ensure you're on a valid version branch**:  

   ```bash
   git checkout 0.5.0        # Valid: 0.5.0, 0.5.0-alpha.1, 2.0.0-rc.1
   ```

2. **Run the script**:  

   ```bash
   ./rbg/tools/release/version-update.sh
   ```

---

### ‚úÖ Requirements  

| Item               | Requirement                         | Example                |
|--------------------|-------------------------------------|------------------------|
| **Branch name**    | Must be a semantic version          | `0.5.0`, `1.2.0-rc.1` |
| **Makefile**       | Must contain `VERSION` variable     | `VERSION ?= v0.5.0`    |
| **File structure** | Helm charts in `deploy/helm/rbgs/`  | -                      |

---

### üîÑ What It Updates  

| File               | Field Updated       | Example Value        | Source                     |
|--------------------|---------------------|----------------------|----------------------------|
| `Chart.yaml`       | `version`          | `0.5.0-alpha.2`      | Branch name (without `v`)  |
| `Chart.yaml`       | `appVersion`       | `0.5.0-abc123`       | Image tag without `v`      |
| `values.yaml`      | `image.tag`        | `v0.5.0-abc123`      | Makefile `VERSION` + Git SHA |

## Kustomize and Manifest Maintenance

### Overview

The project uses **kustomize** to generate `deploy/kubectl/manifests.yaml`, which contains all CRDs and controller deployment resources. This file is used for manifest-based installation.

**Key configuration files:**

| Path | Purpose |
|------|---------|
| `config/crd/bases/*.yaml` | CRD definitions generated by controller-gen |
| `config/crd/patches/webhook_in_*.yaml` | Conversion webhook patches for RoleBasedGroup/RoleBasedGroupSet |
| `config/crd/kustomization.yaml` | Applies webhook patches to CRDs |
| `config/default/kustomization.yaml` | Builds complete manifests (CRDs + RBAC + Deployment) |
| `config/manager/kustomization.yaml` | Controller deployment with image tag |

---

### Generate Manifests

**Use `make build-installer` to generate manifests:**

```bash
# Generate manifests with default TAG (VERSION + Git SHA)
make build-installer

# Generate manifests with specific TAG
make build-installer TAG=v0.7.0
```

This command:
1. Runs `make manifests` to regenerate CRD bases from API types
2. Uses kustomize to build `config/default`
3. Outputs to `deploy/kubectl/manifests.yaml`

**Important:** Always use `make build-installer`, never manually edit `manifests.yaml`.

---

### Conversion Webhook Configuration

The CRDs require **conversion webhook** to support v1alpha1 ‚Üî v1alpha2 conversion. This is critical for backward compatibility.

**Verification:**

```bash
# Check conversion webhook exists (should be 2 entries)
grep -c "conversion:" deploy/kubectl/manifests.yaml
grep -c "strategy: Webhook" deploy/kubectl/manifests.yaml

# Expected output: 2 (for RoleBasedGroup and RoleBasedGroupSet)
```

If missing, the manifests were generated incorrectly. Run `make build-installer` to fix.

---

### Release Workflow

When releasing a new version, use `update-release.sh` script:

1. **Checkout version branch:**
   ```bash
   git checkout 0.7.0  # or v0.7.0-alpha.1
   ```

2. **Run update-release.sh:**
   ```bash
   ./tools/release/update-release.sh
   ```

   This script updates:
   - `deploy/helm/rbgs/Chart.yaml` (version, appVersion)
   - `deploy/helm/rbgs/values.yaml` (image.tag)
   - `config/manager/kustomization.yaml` (newTag)
   - `deploy/kubectl/manifests.yaml` (regenerated with conversion webhook)

3. **Commit changes:**
   ```bash
   git add deploy/helm/rbgs/Chart.yaml \
           deploy/helm/rbgs/values.yaml \
           config/manager/kustomization.yaml \
           deploy/kubectl/manifests.yaml
   git commit -s -m "chore: update manifests for v0.7.0 release"
   ```

4. **Verify manifests in CI:** The `e2e-test-manifest` job validates that:
   - Manifests contain conversion webhook configuration (‚â•2 entries)

---

### CI Validation

The `e2e-test-manifest` workflow validates manifests by:

1. Checking conversion webhook count in committed manifests (‚â•2 required)
2. Applying manifests to Kind cluster
3. Running E2E tests

If validation fails with "missing conversion webhook", ensure:
- `config/crd/kustomization.yaml` has webhook patches enabled
- `make build-installer` or `update-release.sh` was used (not manual editing)
